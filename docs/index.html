License Manager API: Especificación

Introducción

El gestor de licencias es una plataforma que permitirá a los clientes solicitar licencias para sus entornos de forma automatizada. Las peticiones deben ser aprobadas por los gestores de la plataforma para que los usarios puedan generar las licencias.

El funcionamiento se basa en crear pools. Los usuarios pueden solicitar un pool que una vez aprobado podrán usar para generar licencias de forma autónoma hasta consumir el pool o que éste expire.

Un usuario podrá crear grupos en los que registrará sus entornos. A la hora de solicitar un pool de licencias, éste quedará asociado de forma única a un entorno perteneciente al grupo donde se encuentra registrado el entorno. Se implementará también una sistema de permisos para los grupos que permita su administración.

Usuarios

La plataforma permitirá el registro de nuevos usuarios. Para el proceso de registro, se requerirán los siguientes siguientes atributos:

Nombre de usuario

Correo electrónico

Contraseña

El proceso de registro requerirá una verificación por parte del usuario. Para ello se le enviará un correo con un enlace que tendrá que abrir para que se considere válida la cuenta.

El usuario podrá hacer login mientras su correo no haya sido verificado, pero no podrá crear grupos o ser añadido a un grupo.

Existirán los siguientes roles:

  • Usuarios de la plataforma: Son los usuarios que realizarán peticiones de licencias. Estos usuarios pueden pertenecer a varios grupos simultáneamente. Sólo tienen visibilidad de los grupos a los que pertenecen. Dentro de cada grupo, a su vez, los usuarios podrán tener diferentes roles:

    • Usuario estándar: Sólamente tiene permiso para realizar peticiones de nuevas licencias para las organizaciones ya existentes del grupo.

    • Administrador: Además de los permisos de los usuarios estándares, pueden añadir nuevas organizaciones al grupo, así como añadir nuevos usuarios al grupo.

    • Propietarios: Sólo existe uno por cada grupo, es el usuario que creó el grupo. Además de los permisos de los administradores, pueden promover usuarios estándares a administradores o degradar administradores a usuarios estándares.

  • Gestories de la plataforma: Son los usuarios que aprueban las peticiones de licencias. No pertenecen a ningún grupo ni tienen acceso a ninguna funcionalidad de las mencionadas anteriormente. Existirá un usuario por defecto con nombre de usario "admin" y contraseña "admin". Este gestor podrá crear más usuarios gestores y será el único medio para crearlos.

Los usuarios serán capaces de cambiar su nombre, email y contraseña. Al cambiar la contraseña se invalidará su token de sesión inmediatamente.

Grupos

Los usuarios podrán crear grupos en la plataforma. Inicialmente los usuarios recién creados no pertenecerán a ningún grupo. Tendrán la posibilidad de crear un nuevo grupo, en el cual asumirán el rol de propietario o aceptar una invitación a un grupo ya existente.

Para invitar un usuario a un grupo, el usuario que invita deberá tener, al menos priviliegios de administrador sobre el grupo. A la hora de invitar, se pedirá una dirección de correo, si el usuario que es invitado ya pertenece al sistema, éste pasará a formar parte del grupo. En caso de que el usuario no esté registrado se producirá un error.

El identificador del grupo incluirá el nombre del usuario que lo ha creado y el nombre del grupo, esto permite que diferentes usuarios puedan crear grupos con el mismo nombre.

Ejemplo 1. Grupos de usuarios

Si un usuario "Usuario" crea un grupo "Grupo", el identificador del grupo será "Usuario/Grupo". Podrán existir dos grupos llamados "John/Test" y "Doe/Test".

Tipos de grupo

Los grupos pueden ser de dos tipos: Teldat o redborder. El tipo de grupo se configurará a la hora de crearlos. Dependiendo del tipo del grupo, las licencias que se generen serán para unos tipos de sensores u otros.

Modo de licenciamiento

Un grupo puede estar configurado para funcionar en modo global o en modo organización.

  • Modelo global: En este modo las licencias generadas funcionarán para un cluster en concreto. La licencia será válida para todo el cluster. No existe el concepto de organización.

  • Modelo por organizaciones: En este modo el cluster se divide en organizaciones. Las licencias que se generan sólamente serán válidas para una organización, no para todo el cluster.

Licencias

Las licencias están asociadas a un cluster en concreto, es decir, cada contiene una identificación única para el cluster al que está asociado. Tratar de añadir una licencia generada para un cluster diferente dará error. En el caso del licenciamiento por organización, cada licencia estará asociada a una organización perteneciente a un cluster.

Los usuarios pertenecientes a un grupo puede realizar peticiones para los clusters asociados al grupo.

Pool de licencias

Cuando se acepta (por parte de un gestor) una petición, se generará un pool de licencias. Un pool permite a los usuarios generar licencias de forma autónoma. Un pool permite generar licencias con unas restricciones en función del Modo de licenciamiento.

Del pool podrán extraerse licencias hasta que se agote o expire.

Licencias trials

Para cada cluster debe ser posible generar una licencia de prueba válida durante un mes. La licencia no tendrá ningún tipo de limitación durante el mes en el que es válida.

Para cada cluster sólo se permitirá generar una licencia trial.

La plataforma no debe olvidar nunca las organizaciones y clusters aunque hayan sido eliminados por un usuario, puesto que permitiría el abuso de las licencias trials.

Como caso excepcional, las licencias trials no son generadas a través de un pool. Las licencias trials podrán ser para clusters de tipo Redborder o Teldat.

Generar licencia

Usar un pool de licencias para generar una nueva licencia. Esto permite al entorno obtener licencias de forma automática conforme van expirando.

Las api keys que pertenecen a un grupo deben tener un contexto limitado a un cluster para evitar posibles problemas de seguridad ya que un cluster no debe ser capaz de pedir/generar licencias para otro.

Una api key será válida siempre en el ámbito de un cluster, independientemente del modo del Modo de licenciamiento de éste. Será responsabilidad del proceso en el cluster de solicitar las licencias para la organización adecuada.
